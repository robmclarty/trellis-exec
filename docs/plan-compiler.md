# Plan Compiler

The plan compiler converts a `plan.md` file into a structured `tasks.json` that the phase runner can execute. It accepts any well-structured technical plan — whether it's a phased task list or an architectural design document — and decomposes it into implementable phases and tasks.

The compiler uses a deterministic-first approach: if the plan is already formatted with phase headings and task lists, it extracts everything with plain TypeScript parsing. Otherwise, it decomposes the plan via LLM using the spec and guidelines as context.

## Why This Architecture

Plans come in many formats. A plan generated by Trellis's `/plan` skill uses structured section headings (`## S1 — Architecture`, `## S2 — Technology Decisions`), while a plan authored by hand might use `## Phase 1: Scaffolding` with task lists. The compiler handles both.

For plans that match the deterministic parser's expected format, extraction is fast, cheap, and reliable. For everything else, the LLM decomposition path produces high-quality task breakdowns by reading the spec for requirements, the plan for architecture, and the guidelines for conventions.

The LLM enrichment path (Stage 2) handles genuinely ambiguous decisions: "does this task depend on the auth module?" or "is this a test-writing task or an implementation task?" These are judgment calls that require understanding intent, not pattern matching.

This split also makes the system testable. The deterministic parser has no external dependencies. The enricher and decomposer accept function parameters instead of importing the agent launcher directly, so tests pass mocks without touching any LLM.

## Three-Stage Pipeline

```text
plan.md + spec.md + guidelines.md
  │
  ▼
┌──────────────────────────────┐
│  Stage 1: Deterministic Parse │  ← planParser.ts
│  Extracts phases, tasks,      │
│  paths, deps, criteria, types │
│  Flags ambiguous fields       │
└──────────┬───────────────────┘
           │
           ├── success ──┐
           │             ▼
           │  ┌─────────────────────────┐
           │  │  Stage 2: Enrichment     │  ← planEnricher.ts
           │  │  Targeted Haiku calls    │
           │  │  for flagged fields only │
           │  └──────────┬──────────────┘
           │             │
           ├── failure ──┐
           │             ▼
           │  ┌─────────────────────────────┐
           │  │  Stage 3: LLM Decomposition  │  ← compilePlan.ts
           │  │  Spec + plan + guidelines    │
           │  │  → complete TasksJson        │
           │  └──────────┬──────────────────┘
           │             │
           ▼             ▼
     ┌──────────────────────┐
     │  Zod validation       │
     │  Write tasks.json     │
     └──────────────────────┘
```

### Stage 1: Deterministic Parser (`planParser.ts`)

A line-by-line state machine that walks the plan markdown and extracts:

- **Phase boundaries** from numbered headings (`## Phase 1: Scaffolding`, `## 1. Setup`)
- **Tasks** from zero-indent list items (`- Do something` or `1. Do something`)
- **Spec references** by matching `§N` patterns
- **File paths** from backtick-delimited strings that contain `/` or end with a known extension
- **Acceptance criteria** from checkbox items (`- [ ] ...`) and labeled blocks (`Acceptance:`, `Verify:`)
- **Sub-agent type** via keyword classification (test-related → `test-writer`, setup-related → `scaffold`, review-related → `judge`, default → `implement`)
- **Dependencies** inferred from `targetPaths` overlap — if task B touches a file created by task A, B depends on A

When a field can't be resolved cleanly, the parser doesn't guess. It records an `EnrichmentFlag` with the task ID, field name, surrounding context, and the reason enrichment is needed. The most common trigger is ambiguous sub-agent classification (e.g., a task that mentions both "test" and "setup").

The parser returns a `ParseResult`:

```typescript
{
  success: boolean;          // false if no phase boundaries found
  tasksJson: TasksJson | null;
  enrichmentNeeded: EnrichmentFlag[];
  errors: string[];
}
```

If the plan uses a freeform structure (no phase headings matching the expected patterns), the parser returns `success: false` and Stage 3 takes over.

### Stage 2: Plan Enricher (`planEnricher.ts`)

Takes the `ParseResult` and a function `enricher: (prompt: string) => Promise<string>` that wraps `llmQuery`. For a well-structured plan, this typically resolves 5–10 fields.

The logic:

1. If no flags exist, return the `TasksJson` as-is. No LLM call.
2. Build a prompt containing each flag with its task context and the full list of available task IDs.
3. Call the enricher and parse the JSON response.
4. Validate the response shape with Zod, then merge resolved values into the `TasksJson`.

If anything fails — the enricher throws, returns invalid JSON, or the response doesn't match the expected schema — the enricher falls back to safe defaults rather than crashing:

| Field | Default |
|-------|---------|
| `dependsOn` | `[]` (no dependencies) |
| `subAgentType` | `"implement"` |
| `acceptanceCriteria` | `[]` (empty) |

The input `ParseResult` is never mutated; `enrichPlan` works on a `structuredClone`.

### Stage 3: LLM Decomposition (inside `compilePlan.ts`)

When the deterministic parser can't find phase boundaries — meaning the plan is an architectural design document rather than a phased task list — the compiler decomposes the plan via LLM. This is the primary compilation path for plans generated by Trellis and other tools that produce technical design documents.

The decomposition prompt (`buildDecomposePrompt` in `prompts.ts`) receives:

1. **Spec content** (full) — defines what to build: requirements, data model, business rules, success criteria
2. **Plan content** (full) — the technical design to decompose: architecture, technology decisions, interface details
3. **Guidelines content** (optional, full) — coding conventions, directory structure, naming rules

The prompt instructs the LLM to:

- Organize tasks into logical build stages (foundation → data layer → business logic → UI → tests), not mirror the plan's section headings
- Create concrete, implementable tasks with specific file paths derived from the guidelines' directory structure
- Derive acceptance criteria from the spec's success criteria
- Classify sub-agent types and infer dependencies based on the architecture

The response is validated against `TasksJsonSchema`. If validation fails, the compiler throws with a descriptive error.

## Prompt Templates (`prompts.ts`)

Two prompt builders, separated from the compilation logic for clarity and testability.

**`buildEnrichmentPrompt(flags, tasks)`** constructs the Stage 2 prompt. It includes:

- The list of all task IDs in the plan (so the LLM can reference valid dependency targets)
- Each flag with its task ID, field name, reason for flagging, context snippet, and the task's full title and description
- Field-specific type instructions (what shape each field value should take)
- A strict output format instruction requesting bare JSON with no preamble

**`buildDecomposePrompt(planContent, specContent, specRef, planRef, guidelinesContent?, guidelinesRef?)`** constructs the Stage 3 decomposition prompt. It includes:

- The complete spec, plan, and (optionally) guidelines content
- The `TasksJson` schema with field descriptions and ID format rules
- Decomposition rules for phasing, task sizing, sub-agent classification, and dependency inference
- The `specRef`, `planRef`, and (optionally) `guidelinesRef` to embed in the output

Both prompts end with an explicit instruction to respond with only JSON. The response parser also strips markdown code fences as a safety measure, since LLMs sometimes wrap JSON output in triple backticks despite being told not to.

## Compile Pipeline (`compilePlan.ts`)

The top-level `compilePlan(config)` function orchestrates everything:

```typescript
type CompileConfig = {
  planPath: string;           // path to plan.md
  specPath: string;           // path to spec.md
  guidelinesPath?: string;    // path to guidelines.md (optional)
  outputPath: string;         // where to write tasks.json
  agentLauncher: AgentLauncher;  // provides llmQuery
};
```

Steps:

1. Read `plan.md` from disk.
2. Run the deterministic parser.
3. If the parser succeeded, run the enricher (wrapping `agentLauncher.llmQuery` as the enricher function). Patch in `guidelinesRef` if provided.
4. If the parser failed, read spec and guidelines content, then run LLM decomposition.
5. Validate the final `TasksJson` against the Zod schema regardless of which path was taken.
6. Write the result to `outputPath`.
7. Return the `TasksJson`.

The `AgentLauncher` dependency is injected via the config, not imported directly. This means `compilePlan` can be tested with a mock launcher, and the CLI can wire in the real one at runtime.

## CLI Usage

```bash
# Plan already in phase/task format (deterministic parse, no LLM needed)
trellis-exec compile plan.md --spec spec.md

# Architectural plan requiring LLM decomposition
trellis-exec compile plan.md --spec spec.md --guidelines guidelines.md

# With enrichment for ambiguous fields
trellis-exec compile plan.md --spec spec.md --guidelines guidelines.md --enrich
```

## Guidelines Support

Guidelines are optional throughout the pipeline:

- **Compile time**: The `--guidelines` flag provides the path. If present, guidelines content is included in the LLM decomposition prompt and `guidelinesRef` is set on the output `TasksJson`.
- **Run time**: If `tasksJson.guidelinesRef` exists, the guidelines file is copied into the project root alongside the spec file so the phase orchestrator can read it directly.

## Testing

Tests live in `src/compile/__tests__/` and `src/__tests__/`:

- **Parser tests** (`planParser.test.ts`): deterministic parsing of various plan formats
- **Enricher tests** (`planEnricher.test.ts`): flag resolution, fallback defaults, immutability
- **Prompt tests** (`prompts.test.ts`): decompose prompt includes spec/plan/guidelines content, schema fields, and handles optional guidelines
- **CLI tests** (`cli.test.ts`): `--guidelines` flag parsing, default behavior without guidelines
- **Phase runner tests** (`phaseRunner.test.ts`): `buildPhaseContext` includes guidelines reference
