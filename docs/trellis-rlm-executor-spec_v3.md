# Trellis RLM Executor

A phased, RLM-inspired task execution system for running implementation plans generated by the Trellis spec-driven development pipeline. Distributed as both a standalone CLI tool and a Claude Code plugin.

## §1 — Problem Statement

Trellis generates structured implementation plans (`plan.md`) from specs. Today, executing those plans requires a single long-running LLM agent session that holds the entire spec, task list, codebase context, and accumulated changes in one context window. Over long task sequences, this causes context rot: the agent loses track of earlier decisions, re-reads files it already processed, contradicts its own prior work, and produces increasingly degraded output.

The goal is to build a task execution system that:

1. Converts a `plan.md` into a structured, phased `tasks.json`.
2. Executes tasks using an RLM (Recursive Language Model) pattern where each phase gets its own long-running orchestrator with REPL-based context management.
3. Maintains continuity between phases through a deterministic outer loop and shared state, not through LLM memory.
4. Dispatches specialized sub-agents for discrete programming tasks with scoped permissions and model routing for cost/speed.
5. Runs experimental code safely in isolated environments.
6. Supports iterative refinement (Ralph Loop) at the phase boundary, allowing the system to retry, correct, or re-plan without re-running completed work.

The secondary motivation is to create a reusable execution layer that any Trellis-generated spec can feed into, making the full pipeline (spec → plan → tasks → execute) automatable end-to-end.

## §2 — Solution Overview

The system has four layers:

**Layer 1: Plan Compiler.** A two-stage pipeline that reads `plan.md` and produces `tasks.json`. Stage 1 is a deterministic TypeScript parser that extracts phase headings, task list items, spec references, file paths, and file-level dependency inference. Stage 2 is a targeted Haiku enrichment call that fills in only the fields the parser couldn't resolve deterministically (natural-language dependency edges, sub-agent type classification). If the plan's structure is too ambiguous for the parser to find phase boundaries at all, a full Haiku structured-output parse serves as fallback.

**Layer 2: Phase Runner (outer loop with Ralph Loop behavior).** A deterministic CLI script that owns the phase queue and the iterative refinement cycle. It reads `tasks.json`, iterates through phases, launches a phase orchestrator for each, collects the orchestrator's phase report (which incorporates judge feedback), updates shared state, and decides next steps: advance to the next phase, retry the current phase with corrective tasks, or halt. This script is not an LLM. It is a Node script with zero AI dependencies. It subsumes what would otherwise be a separate Ralph Loop script. The phase runner also writes a `trajectory.jsonl` log incrementally for observability.

**Layer 3: Phase Orchestrator (RLM layer).** An LLM agent with a persistent JS REPL session, launched once per phase via `claude` CLI subprocess. It receives the phase's task list, the shared state from prior phases, the handoff briefing, and access to the codebase through REPL helpers (not loaded into context). It works through tasks within the phase, using the REPL to explore files, run code, and dispatch sub-agents. Independent tasks within a phase can run in parallel. The orchestrator's tool knowledge is organized across five dedicated skills (`explore-codebase`, `dispatch-agent`, `verify-work`, `manage-phase`, `quick-query`), keeping its core system prompt thin: just phase context, role description, and REPL session protocol. When the orchestrator's message history approaches context capacity, it self-compacts by summarizing completed work and replacing the history. At phase end, the orchestrator invokes the judge, parses the judge's assessment, synthesizes it with its own execution context, and produces a single coherent phase report. The phase report flows up to the phase runner for decision-making.

**Layer 4: Sub-agents.** Claude Code agent files (`agents/*.md`) dispatched by the phase orchestrator via `claude --agent-file` subprocesses for discrete tasks. Each sub-agent receives a focused context bundle (target files, relevant types, task instructions) and returns a result. Sub-agents have restricted permissions (e.g., a test-writer can only create/modify test files). Different sub-agent types can use different models optimized for cost and speed. Adding a new specialist agent is adding a new `.md` file to the `agents/` directory.

Data flows top-to-bottom: `plan.md` → Plan Compiler → `tasks.json` → Phase Runner → Phase Orchestrator (per phase) → Sub-agents (per task) → check/verify (per task) → judge (per phase) → phase report → Phase Runner decides next step.

## §3 — Data Model

### tasks.json

The output of the plan compiler. This is the canonical task representation consumed by the phase runner.

```jsonc
{
  "specRef": "./specs/feature-x/spec.md",
  "planRef": "./specs/feature-x/plan.md",
  "createdAt": "2026-03-17T00:00:00Z",
  "phases": [
    {
      "id": "phase-1",
      "name": "scaffolding",
      "description": "Project setup, config files, base directory structure",
      "tasks": [
        {
          "id": "task-1-1",
          "title": "Initialize package.json and tsconfig",
          "description": "Detailed natural-language instructions for the task...",
          "dependsOn": [],
          "specSections": ["§7", "§8", "§11"],
          "targetPaths": ["package.json", "tsconfig.json"],
          "acceptanceCriteria": [
            "tsc --noEmit exits 0",
            "npm install exits 0"
          ],
          "subAgentType": "implement",
          "status": "pending"
        },
        {
          "id": "task-1-2",
          "title": "Create directory structure",
          "dependsOn": ["task-1-1"],
          "description": "...",
          "specSections": ["§11"],
          "targetPaths": ["src/"],
          "acceptanceCriteria": ["All directories from §11 exist"],
          "subAgentType": "scaffold",
          "status": "pending"
        }
      ]
    }
  ]
}
```

Field definitions:

- `specRef`, `planRef`: relative paths to the source spec and plan, so the orchestrator can read them programmatically via the REPL.
- `phases[].id`: unique identifier, kebab-case, sequential.
- `phases[].tasks[].dependsOn`: array of task IDs within the same phase that must complete before this task starts. Cross-phase dependencies are implicit (all prior phases complete before the next starts). No circular dependencies.
- `phases[].tasks[].specSections`: which spec sections are relevant. The orchestrator uses these to extract targeted context rather than loading the full spec.
- `phases[].tasks[].targetPaths`: files/directories the task is expected to create or modify. Used for permission scoping and parallel scheduling. **Two tasks with overlapping `targetPaths` are implicitly dependent and will not run in parallel**, even if neither lists the other in `dependsOn`.
- `phases[].tasks[].subAgentType`: which agent file to use from `agents/`. See §6 for the agent registry.
- `phases[].tasks[].status`: `"pending"` | `"in-progress"` | `"complete"` | `"failed"` | `"skipped"`. Updated by the phase runner.

### state.json

Persistent shared state maintained by the phase runner across phases.

```jsonc
{
  "currentPhase": "phase-2",
  "completedPhases": ["phase-1"],
  "modifiedFiles": [
    {
      "path": "src/db/schema.ts",
      "modifiedBy": "task-1-3",
      "changeType": "created"
    }
  ],
  "schemaChanges": [
    {
      "table": "users",
      "action": "created",
      "task": "task-1-3"
    }
  ],
  "phaseReports": [
    {
      "phaseId": "phase-1",
      "status": "complete",
      "summary": "Scaffolding complete. Project structure created per §11. Drizzle schema initialized with users and sessions tables. No deviations from spec.",
      "judgeAssessment": {
        "passed": true,
        "issues": [],
        "suggestions": ["Consider adding index on users.email for lookup performance"]
      },
      "orchestratorNotes": "Judge's index suggestion is valid but deferred — the plan addresses indexing in phase-4 (data-access optimization). No action needed now.",
      "decisionsLog": [
        "Used @fastify/autoload for route registration instead of manual imports, consistent with existing Minga patterns."
      ],
      "issuesLog": [],
      "checkResults": {
        "task-1-1": { "passed": true },
        "task-1-2": { "passed": true }
      }
    }
  ],
  "phaseRetries": {
    "phase-1": 0
  }
}
```

### Phase Report

The phase report is the structured output the orchestrator produces at the end of a phase. It is the primary input the phase runner uses for decision-making.

```jsonc
{
  "phaseId": "phase-1",
  "status": "complete",  // "complete" | "partial" | "failed"
  "summary": "Natural language summary of what happened...",
  "tasksCompleted": ["task-1-1", "task-1-2", "task-1-3"],
  "tasksFailed": [],
  "judgeAssessment": {
    "passed": true,
    "issues": [],
    "suggestions": ["..."]
  },
  "orchestratorAnalysis": "The judge flagged X, but this is expected because... Y issue is genuine and should be addressed via...",
  "recommendedAction": "advance",  // "advance" | "retry" | "halt"
  "correctiveTasks": [],  // populated if recommendedAction is "retry"
  "decisionsLog": ["..."],
  "handoff": "# Phase 1 → Phase 2 Handoff\n..."
}
```

The orchestrator synthesizes the judge's raw assessment with its own execution context (what it tried, what failed, what constraints it encountered) into `orchestratorAnalysis`. It also makes a recommendation to the phase runner about what to do next. The phase runner can follow the recommendation or override it (e.g., enforcing a max retry count).

When the orchestrator hits the turn limit or a fatal error, it should attempt to write a `status: "partial"` phase report with whatever it has: completed tasks, in-progress task state, no judge assessment. The phase runner handles missing phase reports gracefully but prefers partial ones, since they preserve context that would otherwise be lost.

The `handoff` field contains a concise markdown briefing (~2000 tokens max) structured as:

```markdown
# Phase N → Phase N+1 Handoff

## What was built
- [concise list of what exists now]

## Key decisions
- [decisions that affect downstream work]

## File manifest
- [files created/modified with one-line descriptions]

## Judge assessment (summarized)
- [key points from the judge, with orchestrator's interpretation]

## Known issues
- [anything the next phase needs to watch for]

## Next phase context
- [specific notes relevant to what the next phase will do]
```

### trajectory.jsonl

A per-execution append-only log for observability. Written incrementally by the phase runner (one JSON line per event, flushed immediately). This is crash-safe by design since nothing is buffered.

```jsonc
// Each line is one of:
{ "phaseId": "phase-1", "turnNumber": 12, "type": "repl_exec", "input": "readFile('src/index.ts')", "output": "...", "timestamp": "2026-03-17T01:12:34Z", "duration": 45 }
{ "phaseId": "phase-1", "turnNumber": 15, "type": "sub_agent_dispatch", "input": { "agent": "implement", "taskId": "task-1-2" }, "output": { "status": "success" }, "timestamp": "...", "duration": 8200 }
{ "phaseId": "phase-1", "turnNumber": 18, "type": "check_run", "input": "npm run lint && npm test", "output": { "exitCode": 0 }, "timestamp": "...", "duration": 3400 }
{ "phaseId": "phase-1", "turnNumber": 25, "type": "judge_invoke", "input": { "taskIds": ["task-1-1", "task-1-2"] }, "output": { "passed": true }, "timestamp": "...", "duration": 12000 }
```

Event types: `"repl_exec"`, `"sub_agent_dispatch"`, `"check_run"`, `"judge_invoke"`.

## §4 — Plan Compiler

### Two-stage architecture

The plan compiler converts `plan.md` into `tasks.json` through a deterministic-first pipeline.

**Stage 1: Deterministic parser (`planParser.ts`)**

A TypeScript module that parses the structured markdown of a plan generated by Trellis's `/plan` skill. It extracts:

1. **Phase boundaries.** Identified by top-level numbered headings or explicit phase markers. Mapped to `phases[]` entries.
2. **Tasks.** Each sub-item under a phase becomes a task. The parser extracts title, description, and any inline metadata.
3. **Spec references.** `§N` patterns in task descriptions are captured as `specSections`.
4. **File paths.** Backtick-delimited file paths (e.g., `` `src/routes/auth.ts` ``) are captured as `targetPaths`.
5. **File-level dependencies.** If task B's `targetPaths` overlap with files created by task A, B depends on A. This is inferred automatically without explicit plan language.
6. **Ambiguity flags.** Fields the parser can't resolve deterministically are marked with `needsEnrichment: true`. Typical candidates: natural-language dependency edges ("do this after the auth module is ready"), sub-agent type classification for ambiguous tasks.

The parser produces a partial `tasks.json` where most fields are filled and a small number are flagged for enrichment.

**Stage 2: Targeted Haiku enrichment (`planEnricher.ts`)**

A `claude --print --model haiku` call that receives ONLY the flagged fields, not the entire plan. Its job is narrow: resolve the ambiguous dependency edges, classify sub-agent types for unclear tasks, and fill in any missing `acceptanceCriteria` that the parser couldn't extract. For a well-formatted plan, this handles roughly 5-10 fields.

The enricher prompt is structured output (JSON mode) and scoped tightly to avoid hallucination. It receives each flagged field with its surrounding context (the task title, description, and neighboring tasks) and returns the resolved values.

**Stage 3: Fallback**

If Stage 1 fails to identify phase boundaries at all (the plan's structure is too freeform for the parser), fall back to a full Haiku structured-output parse of the entire plan. This should be rare when working with plans produced by Trellis's `/plan` skill, which follows a consistent format.

## §5 — Interface Definitions

### CLI Interface (Phase Runner)

The phase runner is the primary entry point. Interactive mode is the default; `--headless` is opt-in.

```text
trellis-exec run <tasks.json> [options]

Options:
  --phase <id>        Run a specific phase only (default: run all remaining)
  --dry-run           Print what would be executed without running
  --resume            Resume from last incomplete task (reads state.json)
  --check <command>   User-defined check command (default: reads from config)
  --isolation <mode>  "worktree" | "none" (default: "worktree")
  --concurrency <n>   Max parallel sub-agents within a phase (default: 3)
  --model <model>     Override the default orchestrator model
  --max-retries <n>   Max phase retries before halting (default: 2)
  --headless          Run without pausing between phases (default: interactive)
  --verbose           Print REPL interactions to stdout
```

Interactive mode (the default) pauses after each phase, displays the phase report, and prompts:

```text
Phase 1 complete.

[Enter] continue  [r] retry  [s] skip  [q] quit
```

This lets the developer inspect results, review changes, and build trust in the system's output before advancing. `--headless` skips these prompts and runs all phases automatically.

```text
trellis-exec status <tasks.json>

Prints current state: which phases are done, which tasks passed/failed,
phase reports, retry counts.
```

### REPL Helper API (available inside Phase Orchestrator)

These functions are pre-loaded into the JS REPL environment before the orchestrator starts. The orchestrator calls them by writing JS code.

```typescript
// Read a file from the project. Returns string content.
// The orchestrator CANNOT read files by importing them; it must use this function.
readFile(path: string): string

// List directory contents. Returns array of {name, type, size}.
listDir(path: string): Array<{name: string, type: "file" | "dir", size: number}>

// Search files by content (grep-like). Returns matching lines with file paths.
searchFiles(pattern: string, glob?: string): Array<{path: string, line: number, content: string}>

// Read specific sections from the spec by section number.
// Returns the markdown content of those sections only.
readSpecSections(sections: string[]): string

// Dispatch a sub-agent via claude --agent-file subprocess.
// The agent file is resolved from agents/ in the plugin directory.
// The context bundle is assembled from the provided file paths and instructions.
dispatchSubAgent(config: {
  type: string,           // agent name (maps to agents/{type}.md)
  taskId: string,         // for tracking
  instructions: string,   // natural-language instructions
  filePaths: string[],    // files to include in the sub-agent's context
  outputPaths: string[],  // files the sub-agent is allowed to create/modify
  model?: string,         // model override (default from agent frontmatter)
}): Promise<SubAgentResult>

// Run the user-defined check command against the project.
// This runs whatever the user configured (e.g., "npm run lint && npm test").
runCheck(): Promise<CheckResult>

// Read the current shared state.
getState(): SharedState

// Write the phase report (called once, at end of phase).
writePhaseReport(report: PhaseReport): void

// Call a fast LLM for quick analysis (not full task execution).
// Thin wrapper around: claude --print --model <model> "<prompt>"
// Defaults to Haiku (cheapest/fastest). Use the optional model parameter
// to override for harder analysis tasks.
llmQuery(prompt: string, options?: { model?: string }): Promise<string>
```

**`llmQuery()` is a first-class orchestrator tool.** The orchestrator should use it liberally for interpretive work: analyzing check failure output, inspecting file structure, reading and summarizing spec sections, deciding task strategy, and evaluating whether a sub-agent's output meets acceptance criteria. Reserve `dispatchSubAgent()` for actual file creation and modification. The `quick-query` skill documents this distinction with examples.

Output truncation rule: any REPL output exceeding 8192 characters is truncated with a `[TRUNCATED — showing first 8192 chars of N total]` marker. The orchestrator must use programmatic filtering (search, slice, map) to work with large outputs, not print entire files.

**Scaffold restoration rule:** After each JS REPL eval in the orchestrator session, the phase runner restores all helper function references (`readFile`, `listDir`, `searchFiles`, `readSpecSections`, `dispatchSubAgent`, `runCheck`, `getState`, `writePhaseReport`, `llmQuery`, etc.) to their original implementations. This prevents LLM-generated code from accidentally overwriting a helper by assigning to the same variable name.

### Sub-Agent Input/Output Contract

Sub-agents are Claude Code agent files in the `agents/` directory. The phase runner launches them via `claude --agent-file agents/{type}.md` subprocesses. The orchestrator assembles the prompt from the context bundle and dispatches it through the `dispatchSubAgent()` helper.

Sub-agent input (assembled by the orchestrator):

```text
You are a [type] sub-agent. Your task:

[instructions from orchestrator]

You may ONLY create or modify these files:
[outputPaths list]

Here are the files you need to read:
--- src/db/schema.ts ---
[file contents]
--- src/types/user.ts ---
[file contents]

Relevant spec sections:
[spec excerpt]

Respond with the complete contents of each file you create or modify.
```

Sub-agent output is parsed by the orchestrator from the `claude --print` response.

### Verification Tiers

Three tiers of verification, evaluated per task or per phase as noted:

**1. Check (per task, deterministic, required)**

A user-defined shell command configured per project. Runs after each task completes. This is the hard gate. If check fails, the task is not considered complete.

Example configurations:

- `"npm run lint && npm run build && npm test"`
- `"make check"`
- `"./scripts/verify.sh"`

Configured via `--check` CLI flag, a `trellis-exec.config.json` file, or a `check` field in `tasks.json`.

**2. Verify (per task, LLM-generated, optional)**

The orchestrator dynamically generates a verification step based on what the sub-agent just did. This could be a quick assertion, a specific test invocation, or an inspection of the output. The orchestrator decides whether to verify and what to verify. This runs inside the orchestrator's REPL session.

Examples of what the orchestrator might do:

- Read the created file and check it exports the expected functions
- Run a specific test file that covers the changed module
- Check that a new route is registered by grepping the route config
- Invoke `llmQuery()` to ask whether the implementation matches a spec section

**3. Judge (per phase, LLM, optional)**

Invoked by the orchestrator as its final act before producing the phase report. The judge is a sub-agent (`agents/judge.md`) that receives the full set of changes made during the phase and the relevant spec sections. It returns a structured assessment: pass/fail, issues found, suggestions.

The orchestrator parses the judge's response, synthesizes it with its own execution context (what was attempted, what was tricky, what constraints applied), and incorporates both into the phase report. This means the phase runner receives a single coherent analysis rather than two separate signals to reconcile.

The judge does NOT create tasks or modify files. It is read-only. If the judge identifies issues that require corrective work, the orchestrator includes recommended corrective tasks in the phase report. The phase runner decides whether to act on them.

## §6 — Business Logic and Service Layer

### Phase Runner Logic

The phase runner is entirely deterministic. Zero LLM calls. Its loop:

```text
for each phase in tasks.json where status != "complete":
  1. Read state.json (or initialize if first phase)
  2. Read handoff from prior phase report (if exists)
  3. Prepare the phase orchestrator's launch context:
     - Agent file (agents/phase-orchestrator.md)
     - Skills directory (skills/)
     - Phase task list (from tasks.json)
     - Shared state summary (from state.json)
     - Handoff briefing (from prior phase report)
     - Check command config
  4. Start a JS REPL process
  5. Load REPL helper functions into the REPL
  6. Start the phase orchestrator via:
       claude --agent-file agents/phase-orchestrator.md --add-dir skills/
  7. Enter the REPL loop:
     a. Send the orchestrator's code output to the REPL
     b. Execute, capture stdout (truncate to 8192 chars)
     c. Restore all REPL helper function references (scaffold restoration)
     d. Track consecutive errors (halt if 5+ sequential errors, configurable)
     e. Feed truncated output back to the orchestrator
     f. Append trajectory event to trajectory.jsonl
     g. Repeat until orchestrator signals phase complete
        or until a hard turn limit is reached
  8. Read the phase report from the orchestrator
     (If turn limit or fatal error: request partial phase report)
  9. Update state.json with:
     - Completed tasks and their check results
     - Modified files manifest
     - Phase report (including judge assessment + orchestrator analysis)
     - Decisions log
  10. Decide next step based on phase report:

      if report.status == "complete" and report.recommendedAction == "advance":
        commit changes in worktree
        if interactive mode (default): pause, show phase report, prompt:
          [Enter] continue  [r] retry  [s] skip  [q] quit
        proceed to next phase

      else if report.recommendedAction == "retry"
             and retryCount < maxRetries:
        append report.correctiveTasks to the phase's task list
        increment retryCount
        re-enter phase (go to step 1 for same phase)

      else:
        halt, print phase report, wait for human intervention

  11. After all phases complete:
      merge worktree branch back to main
      print final summary
```

### Phase Orchestrator Logic

The orchestrator is a `claude` CLI subprocess launched with `--agent-file agents/phase-orchestrator.md --add-dir skills/`. Its system prompt is thin: just phase context, role description, and REPL session protocol. All tool knowledge lives in the five skills, which are auto-loaded from the `skills/` directory via the `--add-dir` flag.

The orchestrator:

1. Reads the phase's task list from its launch context.
2. Respects dependency ordering: only starts a task when its `dependsOn` tasks are complete.
3. Schedules independent tasks in parallel (up to concurrency limit). **Two tasks with overlapping `targetPaths` are treated as implicitly dependent** and are never scheduled concurrently, even if neither lists the other in `dependsOn`.
4. For each task:
   a. Uses REPL helpers to explore the codebase and understand current state.
   b. Reads relevant spec sections using `readSpecSections()`.
   c. Uses `llmQuery()` for analysis and strategy decisions (interpreting check failures, reading file structure, understanding spec requirements).
   d. Assembles a focused context bundle for the sub-agent.
   e. Dispatches the appropriate sub-agent via `dispatchSubAgent()`.
   f. Applies the sub-agent's file changes to the workspace.
   g. Runs the check command via `runCheck()`.
   h. Optionally runs a dynamic verify step.
   i. If check or verify fails: analyzes the failure (via `llmQuery()`), optionally retries with adjusted instructions (max 3 retries per task), or marks as failed.
   j. Updates task status.
5. After all tasks: invokes the judge sub-agent.
6. Parses the judge's assessment.
7. Synthesizes the judge's findings with its own execution context into a phase report. The orchestrator adds interpretive context: why certain issues exist, whether flagged problems are intentional tradeoffs, and what corrective action it recommends.
8. The phase report includes a recommended action (advance / retry / halt) and, if retry, a set of corrective tasks the orchestrator proposes.
9. Writes the phase report via `writePhaseReport()`.
10. Signals phase complete.

The orchestrator is allowed to reorder independent tasks within the phase if it determines a different ordering would be more efficient. It must not skip tasks or modify tasks from other phases. It CAN create new corrective tasks within the current phase in response to check/verify failures, and CAN recommend corrective tasks for phase-level issues detected by the judge.

### Context Compaction

Adopted from the RLM reference implementation. When the orchestrator's message history approaches ~80% of context capacity, it self-summarizes:

- Completed tasks (IDs, outcomes)
- Remaining tasks (IDs, current status)
- Modified files with one-line descriptions of changes
- Check failures and how they were resolved
- Current task state and what the orchestrator was in the middle of doing
- Key decisions made so far

The summary replaces the message history while preserving the system prompt and phase context. A `history` REPL variable holds the pre-compaction context as a string, accessible if the orchestrator needs to recall specific details. The orchestrator can also call `getState()` to verify its summary against the ground-truth shared state.

This prevents context rot during long phases without losing continuity. The trajectory log provides a complete external record regardless of compaction.

### Consecutive Error Tracking

The phase runner tracks sequential REPL errors within an orchestrator session. If 5 or more consecutive turns produce errors (configurable via `TRELLIS_EXEC_MAX_CONSECUTIVE_ERRORS`), the phase is halted as structurally broken. This is distinct from per-task retries, which handle individual task failures. Consecutive error tracking catches situations where the orchestrator itself is stuck in an error loop (malformed REPL code, misunderstood API, broken assumptions).

When the consecutive error threshold is hit, the phase runner requests a partial phase report before terminating the session.

### Sub-Agent Registry

Sub-agents are Claude Code agent files in `agents/`. Each `.md` file uses frontmatter to declare its configuration:

```yaml
---
name: implement
description: Create or modify source files per task instructions
tools:
  - file_write
  - file_read
model: sonnet
---
```

The v1 registry:

| Agent file | Model | Purpose | Permissions |
|------------|-------|---------|-------------|
| `implement.md` | Sonnet | Create or modify source files | Write to specified `outputPaths` only |
| `test-writer.md` | Sonnet | Create test files | Write to `**/*.test.ts` and `**/*.spec.ts` only |
| `scaffold.md` | Haiku | Generate boilerplate (configs, dirs) | Write to specified paths |
| `judge.md` | Sonnet | Evaluate changes for spec compliance | Read only, no file changes |

Model assignments are defaults, overridable per project via config. Adding a new specialist agent is adding a new `.md` file to `agents/` with appropriate frontmatter. No code changes required.

## §7 — Constraints

### Technical constraints

- **Runtime**: Node.js 22+ (for native TS strip support and modern APIs). Bun as alternative if available.
- **Language**: TypeScript with strict mode. Target ES2022+.
- **Style**: Functional and procedural. No classes except where required by external APIs. Use modules for organization. camelCase for variables and functions. Types with PascalCase. Files with kebab-case.
- **Dependencies**: Minimize, but pragmatic. Prefer built-in APIs (e.g., `fetch` over `axios`, `node:fs` over `fs-extra`). An LLM SDK is acceptable if needed. A full agent framework (LangChain, LlamaIndex, Vercel AI SDK) is not. A glob library for file matching is acceptable. Evaluate each dependency against the question: "does this add significant value over the built-in alternative?"
- **LLM interaction**: All LLM calls go through `claude` CLI subprocesses. This assumes Claude Code is installed in the execution environment. No direct Anthropic API calls in v1. The `claude --print` flag is used for sub-agents and `llmQuery()`. The orchestrator session uses the interactive `claude` interface through the REPL loop.
- **No classes, no inheritance, no decorators.** Use plain objects, functions, and modules.
- **Error handling**: Throw exceptions for failures. Catch at layer boundaries (CLI catches phase runner errors, phase runner catches orchestrator errors). Errors should include enough context to diagnose: what was being attempted, which task/phase, relevant file paths.
- **Pre-built distribution**: The `dist/` directory is committed to the repo. The `package.json` `bin` entry points to `./dist/cli.js`. The release workflow (or a manual `npm run build`) produces `dist/` before tagging. This means the plugin works immediately after installation with no build step.

### Scope constraints

- **No web UI.** This is a CLI tool only.
- **No database.** All state is JSON files on disk.
- **No streaming output** from sub-agents in v1. Sub-agents return complete responses.
- **Single-project scope.** The tool operates on one project at a time. No multi-repo orchestration.
- **No `llmQueryBatched()`** in v1. Concurrent multi-file analysis via batched LLM queries is deferred to v1.1.

### Operational constraints

- Runs locally on the developer's machine. No server deployment.
- Requires Claude Code installed and authenticated.
- Assumes the target project already exists (at minimum, an initialized git repo). The tool does not create projects from scratch; the first phase handles scaffolding within an existing repo.
- Requires git. The isolation strategy uses git worktrees and commits at phase boundaries.

## §9 — Failure Modes

### Phase orchestrator hits turn limit

**Scenario**: The orchestrator's REPL loop exceeds the configured maximum turns (default: 100) without signaling completion.

**Expected behavior**: Phase runner requests a partial phase report from the orchestrator (with `status: "partial"`, whatever tasks are complete, no judge assessment). If the orchestrator can't produce one, the phase runner marks the current task as `failed`, writes a partial state update with everything completed so far, and halts.

**Recovery**: Developer reviews the state, fixes the issue, and resumes with `--resume`.

### Consecutive REPL errors

**Scenario**: The orchestrator produces 5 or more consecutive REPL errors (malformed code, misunderstood helpers, etc.).

**Expected behavior**: Phase runner halts the phase as structurally broken. Requests a partial phase report before terminating. Logs the error sequence to `trajectory.jsonl`.

**Recovery**: Developer reviews the trajectory log, identifies the pattern, and either adjusts the plan or resumes after fixing the issue.

### Sub-agent produces unparseable output

**Scenario**: A `claude --print` sub-agent call returns output the orchestrator can't parse into file changes.

**Expected behavior**: The orchestrator logs the raw output, retries the sub-agent with the same instructions plus a note about the expected output format. After 3 retries, marks the task as failed.

### Check command fails

**Scenario**: The user-defined check command exits non-zero after a sub-agent's changes are applied.

**Expected behavior**: The orchestrator receives the check failure output (stderr/stdout), uses `llmQuery()` to analyze it, and dispatches a corrective sub-agent call with the error context. After 3 correction attempts, marks the task as failed and proceeds to the next independent task (or halts if downstream tasks depend on it).

### Git worktree conflicts

**Scenario**: The worktree can't be created because the branch already exists or the working directory is dirty.

**Expected behavior**: Phase runner detects the conflict before launching the orchestrator. It reports the issue and exits with a clear error message. It does not force-delete branches or stash changes.

### Claude CLI not available

**Scenario**: `claude` is not installed or not authenticated.

**Expected behavior**: Phase runner checks for `claude` availability on startup (before any work begins). If not found, exits immediately with a clear message: "Claude Code CLI is required. Install it with: [install instructions]."

### API rate limiting or network failure

**Scenario**: A `claude` subprocess fails due to rate limiting or network issues.

**Expected behavior**: The orchestrator retries the sub-agent call with exponential backoff (handled by Claude Code internally for most cases). If the `claude` process exits with a non-zero code after its own retries are exhausted, the orchestrator marks the task as failed.

### Parallel task file conflict (should not happen)

**Scenario**: Two parallel tasks attempt to write to the same file simultaneously.

**Expected behavior**: This should be prevented by the scheduler. The orchestrator checks `targetPaths` overlap before scheduling tasks in parallel. If overlap is detected, tasks are serialized. If a conflict somehow occurs at runtime (e.g., a sub-agent writes to an undeclared path), the second write fails and the orchestrator halts parallel execution for the remainder of the phase, falling back to sequential.

### Phase retry limit exceeded

**Scenario**: A phase has been retried `maxRetries` times and still fails or the judge still flags issues.

**Expected behavior**: Phase runner halts and prints the latest phase report with all accumulated context. The developer decides whether to manually fix the issue and `--resume`, or to revise the plan/spec.

### Context compaction data loss

**Scenario**: The orchestrator compacts its context and loses a detail it needs for a later task in the same phase.

**Expected behavior**: The orchestrator can recover by calling `getState()` to read ground-truth shared state, or by using `readFile()` to re-inspect the actual codebase. The `history` REPL variable also holds the pre-compaction context as a fallback. The trajectory log provides a complete external record of all REPL interactions regardless of compaction.

## §10 — Success Criteria

### Automated / verifiable

1. `trellis-exec compile` produces valid `tasks.json` from a sample `plan.md` that matches the schema defined in §3.
2. `trellis-exec run` executes a 2-phase, 4-task plan against a trivial test project (e.g., "create a module with two functions and a test file") and all tasks pass check.
3. State persists correctly: after running phase 1, `state.json` contains the completed tasks, modified files, and phase report. Running with `--resume` picks up at phase 2.
4. Handoff is generated at end of phase 1 (inside the phase report) and consumed by the phase 2 orchestrator without errors.
5. The REPL output truncation works: a `readFile()` call on a file larger than 8192 chars returns truncated output with the truncation marker.
6. Sub-agent permission enforcement: a sub-agent that attempts to write to a path outside its `outputPaths` has that write rejected.
7. `--dry-run` prints the full execution plan (phases, tasks, dependencies, sub-agent types, parallel groups) without making any LLM calls or file changes.
8. Parallel scheduling: given 4 tasks where A→B (dependent) and C, D (independent of each other and of A/B), the orchestrator runs A first, then B+C+D in parallel.
9. `targetPaths` overlap detection: given tasks X targeting `src/foo.ts` and Y targeting `src/foo.ts`, they are serialized even without an explicit `dependsOn`.
10. Phase retry: if the orchestrator returns `recommendedAction: "retry"` with corrective tasks, the phase runner appends those tasks and re-enters the phase. After `maxRetries`, it halts.
11. Context compaction: during a phase with 50+ orchestrator turns, the compaction mechanism triggers and the orchestrator continues working correctly with the summarized context. Post-compaction, `getState()` returns consistent data.
12. Scaffold restoration: after a REPL eval that assigns to `readFile = "oops"`, the next turn still has a working `readFile()` function.
13. Interactive mode: with `--interactive` (the default), the runner pauses after each phase and accepts `Enter` (continue), `r` (retry), `s` (skip), or `q` (quit).
14. Headless mode: with `--headless`, all phases run without interactive prompts.
15. Trajectory logging: after a run, `trajectory.jsonl` contains one JSON line per REPL turn, sub-agent dispatch, check run, and judge invocation, each with a valid timestamp and duration.
16. Consecutive error halt: if the orchestrator produces 5 sequential REPL errors, the phase halts and a partial report is attempted.
17. Partial phase reports: when the turn limit is hit, the orchestrator produces a `status: "partial"` report with completed tasks listed.
18. Plan compiler deterministic stage: given a well-formatted plan with `§N` spec refs and backtick file paths, Stage 1 extracts phases, tasks, specSections, and targetPaths without any LLM call.
19. `llmQuery()` defaults: calling `llmQuery("test")` uses Haiku; calling `llmQuery("test", { model: "sonnet" })` uses Sonnet.

### Architectural validation (code review)

- The phase runner contains zero LLM calls. All LLM interaction is via `claude` CLI subprocesses launched by the orchestrator and plan compiler.
- The REPL helper functions are the only interface between the orchestrator and the filesystem. The orchestrator's system prompt does not include raw file contents.
- Error messages at each layer boundary include task ID, phase ID, and relevant file paths.
- The orchestrator's phase report always includes both the judge's raw assessment and the orchestrator's synthesized analysis. The phase runner never sees the judge's output directly; it's always mediated through the orchestrator.
- Sub-agents are `.md` files in `agents/` with declarative frontmatter. No sub-agent configuration lives in TypeScript code.
- The orchestrator's tool knowledge lives in skills (`skills/`), not in a monolithic system prompt.
- `trajectory.jsonl` is written per-turn, not buffered. It survives process crashes.

## §13 — Open Questions

### Resolved

**Orchestrator system prompt design.** Resolved in v3. The orchestrator's system prompt is thin (phase context, role description, REPL session protocol). Tool knowledge lives in five skills (`explore-codebase`, `dispatch-agent`, `verify-work`, `manage-phase`, `quick-query`) loaded via `--add-dir skills/`. This replaces the monolithic `prompts/orchestrator-system.md` from v2.

**Plan compiler: deterministic vs LLM.** Resolved in v3. Deterministic parser first (Stage 1: phase headings, task items, `§N` refs, backtick paths, file-level dependency inference). Targeted Haiku enrichment second (Stage 2: only flagged ambiguous fields). Full LLM fallback only if the parser can't find phase boundaries at all.

**Interactive vs headless default.** Resolved in v3. Interactive is the default. `--headless` is opt-in. Interactive prompt offers: `[Enter] continue`, `[r] retry`, `[s] skip`, `[q] quit`.

### Remaining

**Sub-agent specialization architecture.** v1 has four sub-agent types: implement, test-writer, scaffold, judge. Future versions should support more specialized agents. The working concept is a two-tier model: a task sub-orchestrator that receives a task, explores the codebase, determines what kind of work is needed, and dispatches specialists. Specialists would be narrow, fast, and cheap: a backend/server specialist, a CLI/bash specialist, a utility/tool maker, a shared-module manager, a frontend specialist.

The sub-orchestrator would act as a routing layer: "this task needs a new Fastify route plus an update to a shared Zod schema, so I'll call the backend specialist for the route and the shared-module manager for the schema update."

This is deferred because the right specialization boundaries aren't clear yet. They should be discovered empirically by running the v1 general-purpose implementer and observing where it falls short. The extension point is now `agents/`: adding a new specialist is adding a new `.md` file with the right frontmatter. No code changes required.

**REPL implementation: node:vm vs child_process.** Two options for the JS REPL: `node:vm` (sandboxed V8 context, fast, easy helper injection, but less isolated) vs `child_process` (separate REPL process, better isolation, more complex IPC). Preliminary leaning: `node:vm` with a timeout per execution (e.g., 30 seconds). The orchestrator is already trusted code, and the simplicity of `node:vm` is worth the tradeoff. Add a watchdog timer that kills the vm context if it exceeds the timeout.

**Cost tracking and budgets.** The system should track token usage and cost per phase and per task. Claude Code may expose usage data that can be captured from subprocess output. This is not designed in v1 but the trajectory log captures invocation metadata (model used, duration) to make cost analysis easy to add later.

**Worktree strategy.** Use a single git worktree for the entire execution run, branched off the current HEAD. Each phase commits its changes within the worktree at the phase boundary. On complete success, the branch is merged back to the main branch. On failure, the developer can inspect the branch, cherry-pick successful phases, or reset. Open: should the worktree branch naming be deterministic (e.g., `trellis-exec/{spec-name}/{timestamp}`) or user-specified? Deterministic is simpler and avoids collisions.

**Batched queries.** `llmQueryBatched()` for concurrent multi-file analysis (e.g., "read these 5 files and tell me which ones export a Router type"). Deferred to v1.1.

**Hooks integration.** The `hooks/hooks.json` file is a placeholder for v1.1. Potential hooks: `pre-phase`, `post-phase`, `pre-task`, `post-task`, `on-failure`. These would allow project-specific automation (e.g., "run database migrations before phase 3") without modifying the executor's core logic.

The executor is published through two channels that share the same `src/` codebase:

**CLI tool (npm)**

npm install -g @robmclarty/trellis-exec
trellis-exec run tasks.json --interactive
trellis-exec compile plan.md --spec spec.md

### Path resolution

When the phase runner script launches `claude` CLI subprocesses, it needs to resolve paths to `agents/*.md` and `skills/` within the plugin directory:

```typescript
const pluginRoot = process.env.CLAUDE_PLUGIN_ROOT ?? resolve(__dirname, "../..")

const agentFile = resolve(pluginRoot, "agents/phase-orchestrator.md")
const skillsDir = resolve(pluginRoot, "skills")
```

`CLAUDE_PLUGIN_ROOT` is set automatically by Claude Code for plugin contexts. The fallback (`resolve(__dirname, "../..")`) handles the direct CLI case where the script runs from `dist/cli.js`, walking up to the repo root where `agents/` and `skills/` live.
